# COBOL→Javaマイグレーション デモ手順書

## デモの目的
サンプルデータを用いて、フレームワークの全Stepを実行し、欠損トレーサビリティを含めた精度向上の効果を検証する。


## 欠損管理ルール

### 欠損ID体系
```
D-[カテゴリ]-[連番]
  A: 業務知識欠損（業務目的、運用条件、業務判断）
  B: 関連プログラム欠損（コピー句、CALL先、上流/下流ファイル）
  C: システム構造欠損（サブシステム間連携、共通マスタ用途）
```

### 伝播ルール
**欠損は補完されるまで全下流成果物に伝播する。**

| 成果物 | 欠損の表現 |
|--------|-----------|
| COBOL仕様書 | `[欠損:D-X-NNN]` タグ＋説明テキスト |
| Javaコード | コメント `// [欠損:D-X-NNN]` ＋ `throw new UnsupportedOperationException("D-X-NNN: 要補完")` |
| Java仕様書 | `[欠損:D-X-NNN]` タグ＋説明テキスト＋「補完後に実装予定」 |

### 欠損サマリ（全成果物の冒頭に配置）
```
【本成果物に含まれる未解決の欠損】
| ID | カテゴリ | 内容 | 影響箇所 | ステータス |
|----|---------|------|---------|-----------|
| D-A-001 | A:業務知識 | 業務目的の詳細 | 1.概要 | 未補完 |
| D-B-003 | B:関連PGM | コピー句CPYRATE未投入 | 料率参照 | 未補完 |
```

### Step 2：教師データから変換ルール抽出

```
[INSCALC.cbl の内容]

このペアを分析して、以下を抽出してください：
1. ソース要素→仕様書項目のマッピングルール
2. 記述の粒度ルール（どこまで詳細に書くか）
3. 用語変換ルール（COBOL用語→業務用語）
4. ソースにない情報の一覧（仕様書で補完されていた項目）
5. 仕様書のセクション構成テンプレート
```

#### O2の期待フォーマット
```
【変換ガイドライン】

■ マッピングルール
- IDENTIFICATION DIVISION → 1.概要（プログラムID、名称）
- FILE SECTION FD → 2.入出力ファイル + 3.レコードレイアウト
- WORKING-STORAGE → 定数・計算用ワークは処理フロー内で説明
- PROCEDURE DIVISION → 4.処理フロー
- エラー処理段落 → 5.エラー処理（一覧表形式）

■ 粒度ルール
- FDのレコード項目は全項目をレイアウト表に展開
- FILLER項目は「予備」として記載
- COMPUTE文は数式として記載

■ 用語変換ルール
- IC-CONTRACT-NO → 契約番号
- IC-SUM-INSURED → 保険金額
  ...

■ ソースにない補完情報（= 欠損候補テンプレート）
- 業務目的（1.3） → カテゴリA
- 処理区分・実行タイミング（1.4, 1.5） → カテゴリA
- 注意事項・前提条件（6章） → カテゴリA/C

■ セクション構成テンプレート
1. 概要（ID/名称/目的/処理区分/実行タイミング）
2. 入出力ファイル
3. レコードレイアウト
4. 処理フロー
5. エラー処理
6. 注意事項・前提条件
```

## 評価観点まとめ

| 評価項目 | 確認内容 | 対応Step |
|---------|---------|---------|
| ルール抽出の品質 | 教師データから妥当なルールが抽出できたか | Step 2 |
| 3層分解の正確性 | 各層の情報は正確か | Step 3 |
| COBOL仕様書の精度 | ホールドアウトとの一致度 | Step 8 |
| 欠損検出の網羅性 | 本来の欠損を正しく指摘できたか | Step 4 → Step 8 |
| 誤補完の防止 | タグなしで誤った情報を書いていないか | Step 8 |
| 欠損伝播の正確性 | 欠損IDがJavaコード・Java仕様書に正しく伝播しているか | Step 5 → Step 6 |
| クロス検証の有効性 | テストケースで矛盾を検出できたか | Step 7 |
| Javaコードの品質 | COBOLの逐語訳ではなくJavaらしい設計になっているか | Step 5 |
