# COBOL→Javaマイグレーション仕様書生成フレームワーク
## ― Claude Codeによる高精度な変換と欠損トレーサビリティ ―

---

## 1. 背景と課題

### 1.1 前提条件
- COBOLソースコードからJavaへのマイグレーションプロジェクト
- 過去のPoC検証で **Input（COBOLソース）とOutput（仕様書）のペア** が存在する
- 今回はCOBOLソースのみがインプット（関連プログラム全量は投入不可の場合がある）
- 複数のCOBOLソース（メインプログラム + CALL先 + COPY句）を一括投入可能
- 最終成果物は **Javaコード** と **Java仕様書**

### 1.2 主要課題
1. COBOLソース単体では情報が**構造的に欠損**する（関連プログラム、業務知識、運用情報）
2. AIが欠損を「それっぽく」補完すると**機能が変わるリスク**がある
3. 人間の補完を待たずに進む必要があるが、欠損を見失ってはいけない
4. 複数プログラム間の依存関係（CALL/COPY句）を正しく解析・反映する必要がある

### 1.3 設計思想
- **欠損は消さない**：全成果物に欠損IDで伝播し、トレース可能にする
- **一発生成しない**：段階分解で誤りの混入を防ぐ
- **逆方向の検証を入れる**：テストケース照合で矛盾を検出する
- **後方互換**：単一ソース投入時は従来フローで動作し、複数ソース時に自動拡張する

### 1.4 設計判断の記録

本フレームワークの設計過程で検討・決定した主な判断を記録する。

#### COBOL仕様書は必要か？
- 3層分解の出力で業務ロジックは日本語化済みのため、**最終納品物としては不要**
- ただし**人間の補完を受け入れる器**として必要。欠損補完後の情報がJavaコード生成の精度に効く
- 位置づけ：納品物ではなく「補完済み中間成果物」

#### COBOL仕様書→Java設計仕様書→Javaコード vs COBOL仕様書→Javaコード→Java仕様書
- **後者（コード先行）を採用**
- 理由①：Java設計仕様書を書いても、コード生成時に乖離が出る。乖離した瞬間に設計仕様書がゴミになる
- 理由②：COBOL仕様書で業務ロジックは確定済み。Javaコード生成に必要なのは業務ロジック＋Java設計方針（ガイドライン）であり、完全な設計仕様書は不要
- 理由③：保守フェーズで必要なのは「実態と一致した仕様書」。動くコードから起こした仕様書のほうが正確

#### 直接変換（COBOL→Java）vs 仕様書経由
- **仕様書経由をメインライン、直接変換をセカンドオピニオン**として併用（ハイブリッド）
- 直接変換のみだとCOBOLっぽいJavaになり保守性が低い
- 仕様書経由なら業務レビューが可能で、Javaらしい設計を適用できる
- 両方の出力を差分比較し、乖離箇所を重点レビュー

#### 教師データの使い方：Few-shot vs ルール抽出
- **ルール抽出を採用**
- Few-shot（ペアをそのまま例示）はコンテキスト長を消費し、ソースが変わると汎化しにくい
- ルール抽出→適用なら再現性が高く、レビュー可能で、ペアがない新規ソースにも対応可能

## 3. 欠損管理ルール

### 3.1 基本原則
**欠損は下流の全成果物に伝播し続ける。補完されるまで消えない。**

AIが欠損を「それっぽく」埋めて進むのが一番危険。気づかないまま本番に行くと機能が変わり業務事故になる。そのため欠損にIDを振り、全成果物をトレースできるようにする。

### 3.2 欠損の発生パターン

全コードを投入できないことが欠損の主要因となる。

| 欠損の原因 | 例 | 補完先 | 抽出方法 |
|-----------|---|--------|---------|
| ソースに書かれていない | 業務目的、運用条件、実行タイミング | 業務担当/運用担当 | AI推論 |
| 他のプログラムを見ないとわからない | コピー句、CALL先、入力ファイルの生成元 | 開発担当/設計者 | 機械的抽出可能 |
| 複数プログラムを横断しないとわからない | サブシステム間データフロー、共通マスタ用途 | アーキテクト | AI推論+機械的抽出 |

### 3.3 欠損ID体系

```
D-[カテゴリ]-[連番]

カテゴリ:
  A: 業務知識欠損（業務目的、運用条件、業務判断）
  B: 関連プログラム欠損（コピー句、CALL先、上流/下流ファイル）
  C: システム構造欠損（サブシステム間連携、共通マスタ用途）
```

### 3.4 カテゴリBの判定ロジック（投入済み/未投入の分岐）

複数ソース・COPY句が投入された環境では、カテゴリBの欠損判定が変わる：

| 参照先 | input/に存在するか | 判定 |
|--------|------------------|------|
| COPY句 | `input/copybooks/` に存在する | **欠損ではない** → 共通資材として分析対象 |
| COPY句 | `input/copybooks/` に存在しない | **欠損（D-B-NNN）** → 従来通り |
| CALL先 | `input/programs/` に存在する | **欠損ではない** → 別プログラムとして分析対象 |
| CALL先 | `input/programs/` に存在しない | **欠損（D-B-NNN）** → 従来通り |
| ファイル物理名 | 他プログラムのFDで定義されている | **欠損ではない** → プログラム間連携として記載 |
| ファイル物理名 | どのプログラムにも定義がない | **欠損（D-B-NNN）** → 上流/下流ファイルとして記録 |

### 3.5 各成果物での欠損表現

**COBOL仕様書：**
```markdown
### 4.2 初期処理
入力ファイルをオープンし、前提ジョブの完了を確認する。
[欠損:D-A-002] 前提ジョブの具体名・完了判定方法は未確認
```

**Javaコード：**
```java
// [欠損:D-A-002] 前提ジョブの完了確認ロジックは未実装
// 補完後に実装が必要
private void validatePreCondition() {
    throw new UnsupportedOperationException("D-A-002: 要補完");
}
```

**Java仕様書：**
```markdown
### 前処理
前提ジョブの完了を確認する。
[欠損:D-A-002] 前提ジョブの具体名・完了判定方法は未確認。
本機能は補完情報の提供後に実装予定。
```

### 3.6 欠損サマリ（全成果物の冒頭に配置）
```
【本成果物に含まれる未解決の欠損】
| ID | カテゴリ | 内容 | 影響箇所 | ステータス |
|----|---------|------|---------|-----------|
| D-A-001 | A:業務知識 | 業務目的の詳細 | 1.概要 | 未補完 |
| D-B-003 | B:関連PGM | コピー句CPYRATE未投入 | 料率参照 | 未補完 |
```

### 3.7 欠損情報レポート（O4-B）の構造

O4-Bは単なる一覧ではなく**補完依頼書**として機能する。カテゴリごとに確認先を明示し、「この一覧をこの人に渡してください」で動けるようにする。

```
■ カテゴリA：業務知識の補完（→ 業務担当/運用担当へ）
■ カテゴリB：関連プログラム情報の補完（→ 開発担当/設計者へ）
■ カテゴリC：システム全体構造の補完（→ アーキテクトへ）
```

カテゴリBは特に機械的に抽出可能（COPY文、CALL文、SELECT文のASSIGN先を洗い出し）で精度が高い。

## 4. 教師データ活用戦略

### 4.1 目的
過去PoCのInput-Outputペアから変換の暗黙ルールを形式知化し、新規ソースに適用する。

### 4.2 抽出するルール
1. ソース要素→仕様書項目のマッピングルール
2. 記述の粒度ルール
3. 用語変換ルール（COBOL用語→業務用語）
4. ソースにない補完情報の一覧（= 欠損候補テンプレート）
5. 仕様書のセクション構成テンプレート

### 4.3 複数教師ペアの選択・統合ルール

教師ペアが複数存在する場合、以下のルールで選択・統合する：

**選択基準（優先度順）：**
1. **処理区分の一致**：バッチ→バッチ、オンライン→オンラインを優先
2. **業務ドメインの近さ**：同一業務領域（契約管理→契約管理 等）を優先
3. **ソース規模の近さ**：ステップ数・行数が近いものを優先

**統合方針：**
- 複数ペアから抽出したルールに矛盾がある場合 → **ペア固有のルールとドメイン共通のルールを分離**
- ドメイン共通ルール → そのまま O2 に記載
- ペア固有ルール → `[教師固有:ペア名]` タグを付与して記載し、適用可否を人間が判断

### 4.4 ホールドアウト検証
過去ペアの一部を検証用に残し、ガイドラインのみで生成した仕様書と正解を比較。ズレがあればガイドラインを修正。

## 5. 3層分解

一発で仕様書を書かせず、3つのレイヤーに分けて段階的に抽出する。

| 層 | 内容 | 欠損検出 |
|---|---|---|
| ① 静的構造層 | FD、WORKING-STORAGE、ファイルI/O一覧 | 未投入のCOPY句・CALL先→カテゴリB |
| ② 制御フロー層 | PERFORMコールツリー、条件分岐、ループ | CALL文のパラメータ不明→カテゴリB |
| ③ 業務ロジック層 | 各段落の日本語化、計算式、エラーハンドリング | 業務的な「なぜ」が読めない箇所→カテゴリA |

各層を出力するたびに人間がレビューして次のフェーズへ進む。

## 6. 複数プログラム対応ガイドライン

### 6.1 依存関係解析の手法

Step 0で以下の情報を機械的に抽出する：

| 抽出対象 | COBOL構文 | 抽出方法 |
|---------|----------|---------|
| COPY句参照 | `COPY {コピー句名}` | DATA DIVISION / PROCEDURE DIVISION 内のCOPY文をスキャン |
| 外部プログラム呼出 | `CALL '{プログラム名}'` / `CALL WS-PGM-NAME` | PROCEDURE DIVISION 内のCALL文をスキャン |
| ファイル物理名 | `SELECT ... ASSIGN TO {物理名}` | ENVIRONMENT DIVISION のSELECT句をスキャン |

### 6.2 処理順序の決定基準

1. **CALL先プログラム**を先に分析する（呼ばれる側の仕様が分かっていないと、呼ぶ側の業務ロジック分析が不正確になる）
2. **共通COPY句**は全プログラムに先行して分析する（Step 3-0）
3. 同一階層のプログラム間に優先度がない場合は、ソースの行数が小さいものから処理する（先に完了した成果物をコンテキストとして活用できる）
4. **循環依存**（AがBをCALL、BがAをCALL）がある場合は人間に判断を仰ぐ

### 6.3 共通資材の扱い

**COPY句が投入された場合：**
- Step 3-0で全COPY句の静的構造を一括分析し、`output/_common/` に保存
- 各プログラムのStep 3-1では、投入済みCOPY句の項目定義を共通資材から参照（欠損にしない）
- Javaコード生成時、COPY句のレコード定義は**共通モデルクラス**として1度だけ生成

**CALL先プログラムが投入された場合：**
- 先にCALL先の仕様書（O4-A）を生成しておく
- CALL元のStep 3-3（業務ロジック分析）で、CALL先の仕様書を参照して連携内容を記載
- Javaコード生成時、CALL先は**サービスクラス**としてDIパターンで注入

### 6.4 プログラム間の成果物参照ルール

| 参照元のStep | 参照できる成果物 | 用途 |
|-------------|----------------|------|
| Step 3-1（静的構造） | 共通資材の静的構造 | COPY句の項目定義参照 |
| Step 3-3（業務ロジック） | 先行プログラムのO4-A | CALL先の処理内容を把握 |
| Step 4（COBOL仕様書） | 先行プログラムのO4-A | プログラム間連携セクションの記載 |
| Step 5（Javaコード） | 先行プログラムのO5 | import関係・DI構成の整合 |

## 7. Javaマイグレーション戦略

### 7.1 ハイブリッドアプローチ
```
【メインライン】COBOL仕様書 + ソース + Java設計方針 → Javaコード
【参照ライン】COBOL → Java 直接変換（セカンドオピニオン）
```
両方の出力を差分比較し、乖離が大きい箇所を重点レビュー。

### 7.2 Java設計方針（コード生成時にガイドラインとして渡す）
- クラス設計ルール（業務単位で再構成、1段落≠1クラス）
- 命名規則
- 例外処理方針
- パッケージ構成
- COBOLのFD→Java I/Oの変換パターン

### 7.3 複数プログラム時のJava設計方針

| COBOL構造 | Java変換先 | 設計意図 |
|----------|----------|---------|
| メインプログラム | メインクラス（エントリポイント） | バッチ起動の単位 |
| CALL先プログラム | サービスクラス（DI注入） | テスタブルな分離 |
| COPY句（レコード定義） | 共通モデルクラス（POJO/Record） | 重複排除・型安全 |
| COPY句（定数定義） | 共通定数クラス（Enum/Constants） | 一元管理 |
| 共有ファイル | 共通Reader/Writerクラス | I/Oの統一 |

### 7.4 Java仕様書の生成方向
**Javaコード → Java仕様書**（コード先行）。動くコードが事実のソースとなるため、仕様書の正確性が担保される。

## 8. Claude Code実行戦略

### 8.1 基本方針
DEMO_GUIDE（手順書）やフレームワーク文書を**そのままClaude Codeに全量投入しない**。コンテキストが溢れるため、**Stepごとにセッションを分けて実行する**。

DEMO_GUIDEは**人間用のオーケストレーション手順書**。各セッションで何を渡して何を回収するかのガイドとして使う。

### 8.2 セッション分割

#### 単一ソースの場合

| セッション | 実行Step | Claude Codeに食わせるもの | 持ち帰り |
|-----------|---------|--------------------------|---------|
| 1 | Step 1-2 | フレームワーク概要（欠損ルール部分）+ 教師ペア | O1, O2 |
| 2 | Step 3 | O1 + O2 + 新規ソース | O3-1, O3-2, O3-3 |
| 3 | Step 4 | O2 + O3-1 + O3-2 + O3-3 + 欠損管理ルール | O4-A, O4-B |
| 4 | Step 5 | O4-A + ソース + Java設計方針 + 欠損伝播ルール | O5 |
| 5 | Step 6 | O5 + 欠損伝播ルール | O6 |
| 6 | Step 7-8 | O4-A + ソース + 正解仕様書 | O7, O8 |

#### 複数ソースの場合

| セッション | 実行Step | Claude Codeに食わせるもの | 持ち帰り |
|-----------|---------|--------------------------|---------|
| 0 | Step 0 | 全ソース + COPY句 | O0 |
| 1 | Step 1-2 | O0 + フレームワーク概要 + 教師ペア | O1, O2 |
| 1.5 | Step 3-0 | COPY句 + O1 | 共通資材 |
| 2〜N | Step 3-7（プログラム単位） | O1 + O2 + 共通資材 + ソース + 先行成果物サマリ | O3〜O7 |
| N+1 | Step 8 | 全O4-A + 正解仕様書 | O8 |

### 8.3 セッション間の受け渡し
- 各セッションのアウトプットはファイルとして保存
- 次のセッションのインプットとしてそのファイルを渡す
- **欠損管理ルール（セクション3）は必要なセッションに毎回渡す**（欠損の表現方法を忘れさせないため）
- **複数ソース時のプログラム切替**：前プログラムの全成果物を渡す必要はない。O4-A（COBOL仕様書）とO5（Javaコード）のサマリで代替可能

### 8.4 プロンプトテンプレート
各セッション用のプロンプトテンプレートはDEMO_GUIDEに記載。実行時はテンプレートにO1〜O6の実際の内容を埋めて渡す。

## 9. クロス検証

生成した仕様書・コードの精度を、テストケースを介して検証する。

**検証ループ：仕様書 → テストケース生成 → ソース照合 → 矛盾検出 → 修正**

欠損に起因する検証不能箇所は明示する。

複数ソースの場合、プログラム間連携のテストケース（CALL元→CALL先のパラメータ受け渡し、共有ファイルの入出力整合）も作成する。

## 10. 評価観点と精度向上の二軸

### 10.1 精度向上の二軸

精度向上は**欠損補完**と**ガイドライン改善**の2つの軸で進む：

| 軸 | 何が上がるか | どこで対応するか | 例 |
|---|---|---|---|
| **欠損補完** | ソースから**読み取れない**情報の精度 | O4-B → 人間が補完 → 次回実行に反映 | 業務目的、実行タイミング、ジョブフロー |
| **ガイドライン改善** | ソースから**読み取れる**情報の記述品質 | O8改善提案 → 次回Step 2で取り込み | 命名スタイル、セクション粒度、分析観点 |

### 10.2 フィードバックループ

```
Step 8（精度評価）
  │
  ├─→ 欠損補完: O4-Bの未補完項目を人間が埋める → 次回実行で欠損が減る
  │
  └─→ ガイドライン改善: O8の改善提案を次回Step 2に投入 → 記述品質が上がる
```

Step 8で特定された改善提案の例：
- 項目ID接頭辞の命名スタイルをルール化する（CT- vs IC- 等）
- セクション構成テンプレートにコード一覧セクションを追加する
- エラー処理の挙動パターン（複数エラー時の処理継続/中断）を分析観点に追加する

### 10.3 評価項目

| 評価項目 | 確認内容 | 対応Step |
|---------|---------|---------|
| ルール抽出の品質 | 教師データから妥当なルールが抽出できたか | Step 2 |
| 3層分解の正確性 | 各層の情報は正確か | Step 3 |
| COBOL仕様書の精度 | ホールドアウトとの一致度 | Step 8 |
| 欠損検出の網羅性 | 本来の欠損を正しく指摘できたか | Step 4 → Step 8 |
| 誤補完の防止 | タグなしで誤った情報を書いていないか | Step 8 |
| 欠損伝播の正確性 | 欠損IDがJavaコード・Java仕様書に正しく伝播しているか | Step 5 → Step 6 |
| クロス検証の有効性 | テストケースで矛盾を検出できたか | Step 7 |
| Javaコードの品質 | COBOLの逐語訳ではなくJavaらしい設計か | Step 5 |
| 依存関係の反映精度 | プログラム間連携がJavaコードに正しく反映されているか | Step 0 → Step 5 |
| 共通資材の重複排除 | COPY句が1度だけ定義され、正しく参照されているか | Step 3-0 → Step 5 |
| ガイドライン改善の有効性 | O8の改善提案が次回実行で記述品質向上に寄与したか | Step 8（2回目以降） |

## 11. 期待される効果

- 段階分解＋ルール駆動による仕様書生成精度の向上

- 複数プログラムの一括投入による欠損の削減（カテゴリBの解消）
- 共通資材の重複排除によるJavaコードの保守性向上
