# COBOL→Javaマイグレーション 実行指示書
## ― Claude Code向け自動実行ガイド ―

あなたはCOBOL→Javaマイグレーションフレームワークを実行するAIエージェントです。
本指示書とフレームワーク文書（cobol-spec-framework.md）に従い、Step 1からStep 8まで順番に実行してください。

---

## 実行ルール

### 基本動作
- 各Stepのアウトプットを**ファイルとして保存**してから次のStepに進むこと
- 各Stepの完了時に**人間にレビューを求めること**（承認されたら次へ進む）
- コンテキストが厳しくなった場合、現在のアウトプットを保存し「ここまでの成果物を保存しました。次のセッションで続きを実行してください」と報告すること

### 欠損管理（最重要ルール）
- **絶対に欠損を推測で埋めないこと**。欠損を「それっぽく」補完すると機能が変わり業務事故になる
- ソースから読み取れない情報には必ず `[欠損:D-X-NNN]` タグを付与すること
- 欠損は下流の全成果物に伝播させること（COBOL仕様書→Javaコード→Java仕様書）
- 欠損IDは以下の体系に従うこと：
  ```
  D-A-NNN: 業務知識欠損（業務目的、運用条件、業務判断）
  D-B-NNN: 関連プログラム欠損（コピー句、CALL先、上流/下流ファイル）
  D-C-NNN: システム構造欠損（サブシステム間連携、共通マスタ用途）
  ```
- 全成果物の冒頭に以下の欠損サマリを配置すること：
  ```
  【本成果物に含まれる未解決の欠損】
  | ID | カテゴリ | 内容 | 影響箇所 | ステータス |
  ```

### 欠損の表現方法（成果物別）

**COBOL仕様書・Java仕様書：**
```markdown
[欠損:D-A-002] 前提ジョブの具体名・完了判定方法は未確認
```

**Javaコード：**
```java
// [欠損:D-A-002] 前提ジョブの完了確認ロジックは未実装
// 補完後に実装が必要
private void validatePreCondition() {
    throw new UnsupportedOperationException("D-A-002: 要補完");
}
```

## 実行手順

### Step 1：前提知識の注入

**インプット：** ドメイン情報（人間から提供、または以下のデフォルトを使用）

**デフォルトコンテキスト：**
```
- COBOL方言: IBM COBOL
- 業務ドメイン: 生命保険 契約管理
- 処理区分: バッチ
- コピー句: なし
- JCL: なし
```

**タスク：**
1. 上記を整理し、不明点があれば [要確認] をつけてO1_context.mdとして保存
2. 人間にレビューを求める

**アウトプット：** `output/O1_context.md`

---

### Step 2：教師データから変換ルール抽出

**インプット：** `teacher/INSCALC.cbl` + `teacher/INSCALC_SPEC.md`

**タスク：**
1. COBOLソースと仕様書のペアを読み込む
2. 以下の観点で変換ルールを抽出する：
   - ソース要素→仕様書項目のマッピングルール
   - 記述の粒度ルール（どこまで詳細に書くか）
   - 用語変換ルール（COBOL用語→業務用語の言い換えパターン）
   - ソースにない情報の一覧（仕様書で補完されていた項目 = 欠損候補テンプレート）
   - 仕様書のセクション構成テンプレート
3. O2_guidelines.mdとして保存
4. 人間にレビューを求める

**アウトプット：** `output/O2_guidelines.md`

---

### Step 3-1：静的構造分析

**インプット：** `input/CONTRENW.cbl` + `output/O1_context.md`

**タスク：**
1. コンテキストを前提にCOBOLソースの静的構造を整理する：
   - DIVISION/SECTION構成
   - ファイル定義（FD）一覧（論理名/物理名/形式/レコード長）
   - 全レコードレイアウト（項目名/PIC/桁数/88レベル含む）
   - WORKING-STORAGE主要データ項目（フラグ/カウンタ/ワーク/定数）
2. **欠損検出（カテゴリB）：** COPY文・CALL文で参照されているが未投入の資材を一覧化
3. O3-1_static.mdとして保存
4. 人間にレビューを求める

**アウトプット：** `output/O3-1_static.md`

---

### Step 3-2：制御フロー分析

**インプット：** `input/CONTRENW.cbl` + `output/O3-1_static.md`

**タスク：**
1. 制御フローを整理する：
   - PERFORMコールツリー（階層構造で）
   - 条件分岐の一覧（IF/EVALUATE、各分岐の条件と遷移先）
   - ループ構造（PERFORM UNTILの終了条件）
2. O3-2_flow.mdとして保存
3. 人間にレビューを求める

**アウトプット：** `output/O3-2_flow.md`

---

### Step 3-3：業務ロジック分析

**インプット：** `input/CONTRENW.cbl` + `output/O3-1_static.md` + `output/O3-2_flow.md`

**タスク：**
1. 各段落の業務ロジックを日本語で記述する。段落ごとに以下を記載：
   - 段落名
   - 入力（参照データ項目）
   - 処理内容（業務的な意味で説明）
   - 出力（更新データ項目）
   - 条件分岐パターン（あれば）
   - 計算式（あれば）
2. O3-3_logic.mdとして保存
3. 人間にレビューを求める

**アウトプット：** `output/O3-3_logic.md`

---

### Step 4：COBOL仕様書生成 + 欠損情報レポート

**インプット：** `output/O2_guidelines.md` + `output/O3-1_static.md` + `output/O3-2_flow.md` + `output/O3-3_logic.md`

**タスク：**
1. 変換ガイドラインのセクション構成テンプレートに従って仕様書を生成する
2. **以下の3段階で情報を分類すること：**
   - **確信度 高**（ソースから確定）：そのまま記載
   - **確信度 中**（推測可能だが確証なし）：[要確認] タグを付与
   - **確信度 低**（ソースからは読み取れない）：[欠損:D-X-NNN] タグを付与
3. **仕様書冒頭に欠損サマリテーブルを配置すること**
4. 欠損情報レポート（補完依頼書）を以下の3カテゴリに分けて作成する：
   - カテゴリA（業務知識）→ 確認先：業務担当/運用担当
   - カテゴリB（関連プログラム）→ 確認先：開発担当/設計者
   - カテゴリC（システム構造）→ 確認先：アーキテクト
5. O4-A_cobol_spec.mdとO4-B_deficiency_report.mdとして保存
6. 人間にレビューを求める（補完できるものがあればここで補完してもらう）

**アウトプット：** `output/O4-A_cobol_spec.md` + `output/O4-B_deficiency_report.md`

---

### Step 5：Javaコード生成

**インプット：** `output/O4-A_cobol_spec.md` + `input/CONTRENW.cbl` + Java設計方針（下記）

**Java設計方針：**
```
- パッケージ: com.insurance.batch.renewal
- 業務単位でクラスを分割（1 COBOL段落 = 1メソッドではない）
- 例外処理: カスタム例外クラスを使用
- I/O: Java NIO + CSVパーサ
- ログ: SLF4J
- テスタブルな設計（DIパターン）
- 命名規則: Javaの標準的なcamelCase
```

**タスク：**
1. COBOL仕様書の業務ロジックをJavaらしい設計に再構成してコードを生成する
2. **O4-Aに含まれる全ての [欠損:D-X-NNN] をJavaコード内にコメントとして伝播すること**
3. **欠損箇所のロジックは `throw new UnsupportedOperationException("D-X-NNN: 要補完")` とすること**
4. **ファイル冒頭に欠損サマリをコメントで記載すること**
5. output/O5_java_code/ 配下に保存
6. 人間にレビューを求める

**アウトプット：** `output/O5_java_code/*.java`

---

### Step 6：Java仕様書生成

**インプット：** `output/O5_java_code/*.java`

**タスク：**
1. Javaコードから仕様書を生成する。以下を含めること：
   - クラス図（クラス名/メソッド/フィールドの一覧）
   - 各クラスの責務
   - 各メソッドの処理概要（入力/処理/出力）
   - 例外処理フロー
2. **コード内の全ての [欠損:D-X-NNN] を仕様書に伝播すること**
3. **冒頭に欠損サマリテーブルを配置すること**
4. O6_java_spec.mdとして保存
5. 人間にレビューを求める

**アウトプット：** `output/O6_java_spec.md`

---

### Step 7：クロス検証

**インプット：** `output/O4-A_cobol_spec.md` + `input/CONTRENW.cbl`

**タスク：**
1. COBOL仕様書をもとにテストケースを作成する：
   - 正常系：各更新パターン（自動更新/手動更新）
   - 正常系：各満期パターン（未払い/更新不可/回数上限）
   - 異常系：各エラーパターン
   - 境界値：満期判定日の境界（60日ちょうど/59日/61日）
2. 各テストケースが元のCOBOLソースのロジックで期待通りに処理されるか照合する
3. 矛盾があれば、仕様書とソースのどちらに問題があるか指摘する
4. **[欠損] に起因する検証不能箇所は明示すること**
5. O7_cross_validation.mdとして保存
6. 人間にレビューを求める

**アウトプット：** `output/O7_cross_validation.md`

---

### Step 8：精度評価

**インプット：** `output/O4-A_cobol_spec.md` + `holdout/CONTRENW_SPEC.md`

**⚠️ holdout/CONTRENW_SPEC.md はこのStepで初めて読むこと。それ以前に参照してはならない。**

**タスク：**
1. AI生成版と正解版を比較して精度評価レポートを作成する
2. 評価観点：
   - セクション構成の一致度
   - 記載項目の過不足
   - 業務ロジック記述の正確性
   - 用語・表現の一致度
   - 欠損検出の精度（O4-Bで指摘した欠損が正解にあるか）
   - 見逃し欠損（O4-Bで指摘できなかったが正解にある情報）
   - 誤補完（[要確認]タグなしで誤った情報を書いていないか）
3. 以下のスコアを算出すること：
   ```
   - セクション構成一致度: ○/△/×
   - 項目網羅率: XX%
   - ロジック正確性: XX%
   - 欠損検出率: XX%
   - 見逃し率: XX%
   - 誤補完件数: N件
   ```
4. 改善提案（ガイドラインの修正点）を記載する
5. O8_evaluation.mdとして保存
6. 人間に最終報告する

**アウトプット：** `output/O8_evaluation.md`

---

## 完了条件

全てのアウトプットファイル（O1〜O8）が生成され、以下が確認できること：
- 欠損IDが O4-A → O5 → O6 に正しく伝播している
- 欠損サマリが全成果物の冒頭に配置されている
- [要確認] タグなしで推測情報が紛れ込んでいない
- 精度評価レポート（O8）で改善点が明確になっている
