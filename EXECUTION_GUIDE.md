# COBOL→Javaマイグレーション 実行指示書
## ― Claude Code向け自動実行ガイド ―

あなたはCOBOL→Javaマイグレーションフレームワークを実行するAIエージェントです。
本指示書とフレームワーク文書（cobol-spec-framework.md）に従い、Step 0からStep 8まで順番に実行してください。

---

## 実行ルール

### 基本動作
- 各Stepのアウトプットを**ファイルとして保存**してから次のStepに進むこと
- 各Stepの完了時に**人間にレビューを求めること**（承認されたら次へ進む）
- コンテキストが厳しくなった場合、現在のアウトプットを保存し「ここまでの成果物を保存しました。次のセッションで続きを実行してください」と報告すること

### 単一ソース / 複数ソースの判定
- `input/programs/` 配下のソースが **1本のみ** の場合：Step 0をスキップし、従来の単一ソースフローで実行する
- `input/programs/` 配下のソースが **2本以上** の場合：Step 0から開始し、プログラム単位で Step 3〜7 を繰り返す
- `input/copybooks/` にCOPY句が存在する場合：Step 3-1 の前に共通資材分析を実施する

### 欠損管理（最重要ルール）
- **絶対に欠損を推測で埋めないこと**。欠損を「それっぽく」補完すると機能が変わり業務事故になる
- ソースから読み取れない情報には必ず `[欠損:D-X-NNN]` タグを付与すること
- 欠損は下流の全成果物に伝播させること（COBOL仕様書→Javaコード→Java仕様書）
- 欠損IDは以下の体系に従うこと：
  ```
  D-A-NNN: 業務知識欠損（業務目的、運用条件、業務判断）
  D-B-NNN: 関連プログラム欠損（コピー句、CALL先、上流/下流ファイル）
  D-C-NNN: システム構造欠損（サブシステム間連携、共通マスタ用途）
  ```
- 全成果物の冒頭に以下の欠損サマリを配置すること：
  ```
  【本成果物に含まれる未解決の欠損】
  | ID | カテゴリ | 内容 | 影響箇所 | ステータス |
  ```

### 欠損の表現方法（成果物別）

**COBOL仕様書・Java仕様書：**
```markdown
[欠損:D-A-002] 前提ジョブの具体名・完了判定方法は未確認
```

**Javaコード：**
```java
// [欠損:D-A-002] 前提ジョブの完了確認ロジックは未実装
// 補完後に実装が必要
private void validatePreCondition() {
    throw new UnsupportedOperationException("D-A-002: 要補完");
}
```

```

│   ├── INSCALC.cbl            # COBOLソース：保険料計算バッチ
```

## 実行手順

### Step 0：依存関係解析と処理計画（複数ソース時のみ）

> **⚠️ input/programs/ に2本以上のソースがある場合のみ実行。1本の場合はStep 1へスキップ。**

**インプット：** `input/programs/*.cbl` + `input/copybooks/*`（あれば）

**タスク：**
1. `input/programs/` 配下の全ソースをスキャンし、以下を抽出する：
   - 各プログラムの PROGRAM-ID
   - COPY文の参照先一覧
   - CALL文の呼び出し先一覧
   - SELECT文の ASSIGN先（ファイル物理名）
2. 依存関係マップを作成する：
   - プログラム間の CALL 関係（有向グラフ）
   - プログラム→COPY句の参照関係
   - 共有ファイル（複数プログラムが同一ファイルを参照するケース）
3. 処理順序を決定する：
   - **依存される側（CALL先、共通COPY句）を先に分析**
   - 循環依存がある場合はその旨を記載し、人間に判断を仰ぐ
4. 投入済み/未投入の資材を判定する：
   - `input/programs/` に存在するCALL先 → 投入済み（後続Stepで分析対象）
   - `input/copybooks/` に存在するCOPY句 → 投入済み（共通資材として分析）
   - いずれにも存在しない参照先 → 欠損候補（カテゴリB）
5. O0_dependency_map.md として保存
6. 人間にレビューを求める（処理順序の確認）

**アウトプット：** `output/O0_dependency_map.md`

---

### Step 1：前提知識の注入

**インプット：** ドメイン情報（人間から提供、または以下のデフォルトを使用）

**デフォルトコンテキスト：**
```
- COBOL方言: IBM COBOL
- 業務ドメイン: 生命保険 契約管理
- 処理区分: バッチ
- コピー句: なし（input/copybooks/ が空の場合）/ あり（ファイル一覧を記載）
- JCL: なし
```

**タスク：**
1. 上記を整理し、不明点があれば [要確認] をつけてO1_context.mdとして保存
2. 複数ソースの場合、O0_dependency_map.md の概要（プログラム数、依存関係の要約）もコンテキストに含める
3. 人間にレビューを求める

**アウトプット：** `output/O1_context.md`

---

### Step 2：教師データから変換ルール抽出

**インプット：** `teacher/` 配下の全教師ペア

**教師データの選択ルール（複数ペアがある場合）：**
1. 全ペアを読み込み、各ペアの特徴を整理する（処理区分、業務ドメイン、ソース規模）
2. 新規インプットの特徴と**最も類似する**ペアを優先的に分析する
   - 類似度の判断基準：処理区分（バッチ/オンライン） > 業務ドメイン > ソース規模
3. 複数ペアから抽出したルールに矛盾がある場合、**ペア固有のルールとドメイン共通のルールを分離**し、共通ルールを採用する
4. ペア固有のルールは [教師固有:ペア名] タグを付与して記載する

**タスク：**
1. COBOLソースと仕様書のペアを読み込む
2. 以下の観点で変換ルールを抽出する：
   - ソース要素→仕様書項目のマッピングルール
   - 記述の粒度ルール（どこまで詳細に書くか）
   - 用語変換ルール（COBOL用語→業務用語の言い換えパターン）
   - ソースにない情報の一覧（仕様書で補完されていた項目 = 欠損候補テンプレート）
   - 仕様書のセクション構成テンプレート
3. O2_guidelines.mdとして保存
4. 人間にレビューを求める

**アウトプット：** `output/O2_guidelines.md`

---

### Step 3-0：共通資材分析（COPY句が投入されている場合のみ）

> **⚠️ input/copybooks/ にファイルが存在する場合のみ実行。なければStep 3-1へスキップ。**

**インプット：** `input/copybooks/*` + `output/O1_context.md`

**タスク：**
1. 投入された全COPY句の静的構造を分析する：
   - 各COPY句の項目名/PIC/桁数/88レベル
   - どのプログラムから参照されているか（Step 0の情報を利用、または各ソースのCOPY文を確認）
2. 共通資材としての役割を推定する（レコード定義/定数定義/共通ワーク領域 等）
3. `output/_common/O3-1_copybooks_static.md` として保存
4. 人間にレビューを求める

**アウトプット：** `output/_common/O3-1_copybooks_static.md`

---

### Step 3-1：静的構造分析

**インプット：** 対象プログラムの `.cbl` + `output/O1_context.md` + `output/_common/O3-1_copybooks_static.md`（あれば）

> **複数ソースの場合：** Step 0で決定した処理順序に従い、プログラムごとに Step 3-1〜3-3、Step 4〜7 を繰り返す。各プログラムの成果物は `output/{プログラム名}/` 配下に保存する。

**タスク：**
1. コンテキストを前提にCOBOLソースの静的構造を整理する：
   - DIVISION/SECTION構成
   - ファイル定義（FD）一覧（論理名/物理名/形式/レコード長）
   - 全レコードレイアウト（項目名/PIC/桁数/88レベル含む）
   - WORKING-STORAGE主要データ項目（フラグ/カウンタ/ワーク/定数）
2. **欠損検出（カテゴリB）：**
   - COPY文で参照されているが `input/copybooks/` に**未投入**の資材を一覧化 → 欠損
   - COPY文で参照されているが `input/copybooks/` に**投入済み**の資材 → 共通資材分析結果を参照（欠損ではない）
   - CALL文で参照されているが `input/programs/` に**未投入**のプログラムを一覧化 → 欠損
   - CALL文で参照されているが `input/programs/` に**投入済み**のプログラム → 別途分析対象（欠損ではない）
3. 保存先：
   - 単一ソース → `output/O3-1_static.md`
   - 複数ソース → `output/{プログラム名}/O3-1_static.md`
4. 人間にレビューを求める

**アウトプット：** `output/O3-1_static.md` or `output/{プログラム名}/O3-1_static.md`

---

### Step 3-2：制御フロー分析

**インプット：** 対象プログラムの `.cbl` + 対応する `O3-1_static.md`

**タスク：**
1. 制御フローを整理する：
   - PERFORMコールツリー（階層構造で）
   - 条件分岐の一覧（IF/EVALUATE、各分岐の条件と遷移先）
   - ループ構造（PERFORM UNTILの終了条件）
   - CALL文による外部プログラム呼び出し（パラメータ一覧を含む）
2. 保存先：
   - 単一ソース → `output/O3-2_flow.md`
   - 複数ソース → `output/{プログラム名}/O3-2_flow.md`
3. 人間にレビューを求める

**アウトプット：** `output/O3-2_flow.md` or `output/{プログラム名}/O3-2_flow.md`

---

### Step 3-3：業務ロジック分析

**インプット：** 対象プログラムの `.cbl` + 対応する `O3-1_static.md` + `O3-2_flow.md`

**タスク：**
1. 各段落の業務ロジックを日本語で記述する。段落ごとに以下を記載：
   - 段落名
   - 入力（参照データ項目）
   - 処理内容（業務的な意味で説明）
   - 出力（更新データ項目）
   - 条件分岐パターン（あれば）
   - 計算式（あれば）
   - 外部プログラム連携（CALL先が投入済みの場合、そのプログラムの仕様を参照して記載）
2. 保存先：
   - 単一ソース → `output/O3-3_logic.md`
   - 複数ソース → `output/{プログラム名}/O3-3_logic.md`
3. 人間にレビューを求める

**アウトプット：** `output/O3-3_logic.md` or `output/{プログラム名}/O3-3_logic.md`

---

### Step 4：COBOL仕様書生成 + 欠損情報レポート

**インプット：** `O2_guidelines.md` + 対応する `O3-1_static.md` + `O3-2_flow.md` + `O3-3_logic.md`
- 複数ソースの場合、先行プログラムの仕様書（O4-A）も参照可能

**タスク：**
1. 変換ガイドラインのセクション構成テンプレートに従って仕様書を生成する
2. **以下の3段階で情報を分類すること：**
   - **確信度 高**（ソースから確定）：そのまま記載
   - **確信度 中**（推測可能だが確証なし）：[要確認] タグを付与
   - **確信度 低**（ソースからは読み取れない）：[欠損:D-X-NNN] タグを付与
3. **仕様書冒頭に欠損サマリテーブルを配置すること**
4. 複数ソースの場合、プログラム間連携セクションを追加する：
   - CALL元/CALL先の関係
   - パラメータ一覧
   - 共有ファイルの入出力関係
5. 欠損情報レポート（補完依頼書）を以下の3カテゴリに分けて作成する：
   - カテゴリA（業務知識）→ 確認先：業務担当/運用担当
   - カテゴリB（関連プログラム）→ 確認先：開発担当/設計者
   - カテゴリC（システム構造）→ 確認先：アーキテクト
6. 保存先：
   - 単一ソース → `output/O4-A_cobol_spec.md` + `output/O4-B_deficiency_report.md`
   - 複数ソース → `output/{プログラム名}/O4-A_cobol_spec.md` + `output/{プログラム名}/O4-B_deficiency_report.md`
7. 人間にレビューを求める（補完できるものがあればここで補完してもらう）

**アウトプット：** `O4-A_cobol_spec.md` + `O4-B_deficiency_report.md`

---

### Step 5：Javaコード生成

**インプット：** 対応する `O4-A_cobol_spec.md` + 対象プログラムの `.cbl` + Java設計方針（下記）

**Java設計方針：**
```
- パッケージ: com.insurance.batch.{業務名}（プログラムに応じて決定）
- 業務単位でクラスを分割（1 COBOL段落 = 1メソッドではない）
- 例外処理: カスタム例外クラスを使用
- I/O: Java NIO + CSVパーサ
- ログ: SLF4J
- テスタブルな設計（DIパターン）
- 命名規則: Javaの標準的なcamelCase
```

**複数ソース時の追加方針：**
```
- CALL先プログラム → サービスクラスとして生成し、メインプログラムからDIで注入
- 共通COPY句 → 共通モデルクラス（output/_common/ 配下または共通パッケージ）として生成
- プログラム間で共有するファイル → 共通のReader/Writerクラスを生成
- 先行プログラムのJavaコード（O5）が存在する場合、import関係を整合させる
```

**タスク：**
1. COBOL仕様書の業務ロジックをJavaらしい設計に再構成してコードを生成する
2. **O4-Aに含まれる全ての [欠損:D-X-NNN] をJavaコード内にコメントとして伝播すること**
3. **欠損箇所のロジックは `throw new UnsupportedOperationException("D-X-NNN: 要補完")` とすること**
4. **ファイル冒頭に欠損サマリをコメントで記載すること**
5. 保存先：
   - 単一ソース → `output/O5_java_code/`
   - 複数ソース → `output/{プログラム名}/O5_java_code/`
6. 人間にレビューを求める

**アウトプット：** `O5_java_code/*.java`

---

### Step 6：Java仕様書生成

**インプット：** 対応する `O5_java_code/*.java`

**タスク：**
1. Javaコードから仕様書を生成する。以下を含めること：
   - クラス図（クラス名/メソッド/フィールドの一覧）
   - 各クラスの責務
   - 各メソッドの処理概要（入力/処理/出力）
   - 例外処理フロー
   - 複数ソースの場合：クラス間の依存関係（import関係、DI構成）
2. **コード内の全ての [欠損:D-X-NNN] を仕様書に伝播すること**
3. **冒頭に欠損サマリテーブルを配置すること**
4. 保存先：
   - 単一ソース → `output/O6_java_spec.md`
   - 複数ソース → `output/{プログラム名}/O6_java_spec.md`
5. 人間にレビューを求める

**アウトプット：** `O6_java_spec.md`

---

### Step 7：クロス検証

**インプット：** 対応する `O4-A_cobol_spec.md` + 対象プログラムの `.cbl`

**タスク：**
1. COBOL仕様書をもとにテストケースを作成する：
   - 正常系：主要な処理パターン
   - 異常系：各エラーパターン
   - 境界値：判定条件の境界
   - 複数ソースの場合：プログラム間連携のテストケース（CALL元→CALL先のパラメータ受け渡し）
2. 各テストケースが元のCOBOLソースのロジックで期待通りに処理されるか照合する
3. 矛盾があれば、仕様書とソースのどちらに問題があるか指摘する
4. **[欠損] に起因する検証不能箇所は明示すること**
5. 保存先：
   - 単一ソース → `output/O7_cross_validation.md`
   - 複数ソース → `output/{プログラム名}/O7_cross_validation.md`
6. 人間にレビューを求める

**アウトプット：** `O7_cross_validation.md`

---

### Step 8：精度評価

**インプット：** 全プログラムの `O4-A_cobol_spec.md` + `holdout/` 配下の正解仕様書

**⚠️ holdout/ 配下のファイルはこのStepで初めて読むこと。それ以前に参照してはならない。**

**タスク：**
1. AI生成版と正解版を比較して精度評価レポートを作成する
2. 評価観点：
   - セクション構成の一致度
   - 記載項目の過不足
   - 業務ロジック記述の正確性
   - 用語・表現の一致度
   - 欠損検出の精度（O4-Bで指摘した欠損が正解にあるか）
   - 見逃し欠損（O4-Bで指摘できなかったが正解にある情報）
   - 誤補完（[要確認]タグなしで誤った情報を書いていないか）
   - 複数ソースの場合：プログラム間連携の記述精度
3. 以下のスコアを算出すること：
   ```
   - セクション構成一致度: ○/△/×
   - 項目網羅率: XX%
   - ロジック正確性: XX%
   - 欠損検出率: XX%
   - 見逃し率: XX%
   - 誤補完件数: N件
   ```
4. 改善提案（ガイドラインの修正点）を記載する
5. O8_evaluation.mdとして保存（`output/` 直下、全プログラム横断）
6. 人間に最終報告する

- コンテキストが厳しい場合、前プログラムの成果物はサマリ（要約）で代替可能
